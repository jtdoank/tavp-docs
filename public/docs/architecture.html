Architecture Overview

Request Lifecycle

When a request enters TAVP, it follows this lifecycle:

  1. Entry Point — public/index.php bootstraps the application
  2. DI Container — Services are registered and resolved
  3. Middleware Pipeline — Request passes through middleware stack
  4. Router — Request is matched to a route
  5. Controller — Route handler executes business logic
  6. Response — Response is returned to the client

Service Container

TAVP uses Phalcon's DI container as its service container. Services are registered in bootstrap/app.php:

$di->setShared('router', function () {
    $router = new Router();
    require config_path('routes.php');
    return $router;
});

Middleware Pipeline

Middleware runs in order, each can modify the request or response:

$middleware = [
    \Tavp\Middleware\TrustProxies::class,
    \Tavp\Middleware\HandleCors::class,
    \Tavp\Middleware\ValidateCsrfToken::class,
    \Tavp\Middleware\StartSession::class,
    \Tavp\Middleware\Authenticate::class,
];

Three Runtimes

TAVP supports three runtime modes:

  • PHP-FPM — Traditional, works everywhere
  • TAVP Coil (Swoole) — Coroutine-based, highest performance
  • TAVP Relay (RoadRunner) — Go-based, balanced approach
